# Dependency And Compatibility Policy

## Selection Rules

- Prefer latest compatible stable versions.
- Avoid prerelease, nightly, beta, canary, or `0.x` packages unless the project explicitly accepts that risk.
- Check official documentation, release notes, migration guides, compatibility matrices, and known issues.
- Validate with local tests and runtime checks.
- Verify maintenance health, license, security posture, and ecosystem fit for new production dependencies.

## Upgrade Rules

- Patch and minor upgrades still require test evidence.
- Major upgrades require a migration plan and affected-scope analysis.
- Lockfile changes must be generated by the correct package manager.
- Do not mix unrelated dependency upgrades with feature work unless the feature requires them.
- If a dependency introduces broad churn, split the upgrade into its own commit or PR.

## Compatibility Checks

Check compatibility across:

- Runtime version.
- Framework version.
- Language version.
- Build tool and bundler.
- Type system and generated types.
- Peer dependencies.
- Plugin ecosystem.
- Deployment environment.
- Security scanners and policy.

## May 2026 Supply Chain Standards

### SBOM (Software Bill of Materials)

Attach SPDX or CycloneDX SBOM to every release artifact. Generate via:

- npm: `npm sbom --sbom-format cyclonedx > sbom.json`
- Python: `cyclonedx-py environment > sbom.json`
- Cargo: `cargo cyclonedx`
- Go: `cyclonedx-gomod app -output sbom.json`

### SLSA Level 2 (minimum for new repositories)

Verifiable build provenance via GitHub Actions + cosign / `slsa-github-generator`. Achievable in one day:

```yaml
permissions:
  id-token: write
  contents: read
  attestations: write
```

Use `actions/attest-build-provenance@v1` to attest builds. Verify with `gh attestation verify` or `cosign verify-attestation`.

### Sigstore / cosign

Sign release artifacts with `cosign sign --keyless`. Verify upstream signatures when consuming critical dependencies.

### OWASP Top 10 2025 - A03 Software Supply Chain Failures

Top threats in May 2026:

- Dependency confusion (typosquatting).
- Upstream infrastructure compromise.
- Code signing certificate theft.
- CI/CD exploitation (poisoned workflows).
- Untrusted scripts in install hooks.

Defenses: SBOM + SLSA + Sigstore + lockfile discipline + dependency-review-action in CI.

### Lockfile Discipline

- `--frozen-lockfile` / `--immutable` / `npm ci` in CI to enforce exact version matching.
- Commit lockfiles for application projects. For libraries, lockfile commit is optional (test in CI matrix).

## Package Manager Defaults (May 2026)

- **JavaScript/TypeScript**: pnpm (faster, better disk usage, ecosystem-supported).
- **Python**: uv (Astral) for environments + dependency resolution.
- **Rust**: cargo with Cargo.lock committed for binaries.
- **Go**: go mod tidy + go.sum committed.
- **Dart/Flutter**: pubspec.lock committed for application projects.

## Rejection Criteria

Reject or ask before using a dependency when:

- It is abandoned or poorly maintained (no commits in 12+ months without explanation).
- It duplicates a stable project utility.
- It requires broad architecture changes unrelated to the task.
- It hides critical behavior behind magic that is hard to debug.
- Its license or security posture is unclear.
- Its install scripts run untrusted binaries (`postinstall` exploitation vector).
